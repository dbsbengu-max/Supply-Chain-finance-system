## 4. 单证状态与复核流

### 4.1 单证生命周期（简化）

```text
UPLOADED → [OCR] → OCR_COMPLETED / OCR_FAILED
         → [submit-review] → PENDING_REVIEW
PENDING_REVIEW → [approve] → APPROVED（review_status=APPROVED）
               → [reject + reason] → REJECTED
任意非 ARCHIVED → [archive + reason] → ARCHIVED（归档后不可再次 submit-review，409）
```

### 4.2 校验接口逻辑

`POST /documents/validate` 按 `business_type + business_stage (+ product_type)` 匹配必备规则：

- **missing**：必备单证未登记；
- **pending**：已登记但未 OCR 完成 / 未人工复核通过；
- **warnings**：OCR 置信度低于规则 `min_confidence`（默认 0.85）；
- **passed**：无 missing 且无 blocking pending。

## 8. 实现备注

- **迁移 FK 修正**：种子数据无 `ROLE_PLATFORM_OPERATOR`，角色授权仅使用已存在角色，避免 Flyway 外键失败。
- **review log 快照**：`ocr_job_id` / `ocr_confidence` 可为空，快照使用 `HashMap` 构建，避免 `Map.of` NPE。
- **低置信度测试**：Mock OCR 平均置信度约 0.868，专项用例通过 JDBC 将 `ocr_confidence` 设为 0.70 以触发 warning。
- **跨项目策略**：与 EA-019 一致，Header 项目与 JWT 项目不一致时返回 `AUTH_403`，不泄露单证存在性。
- **Codex复核补丁**：`/documents/validate` 查询单证时补充 `operator_id + project_id` 过滤；FINANCE 业务对象补充融资数据范围校验，避免跨项目单证污染校验结果。
