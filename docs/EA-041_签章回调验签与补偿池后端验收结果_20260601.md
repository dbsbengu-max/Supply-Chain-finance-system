## 3. 验签契约

`TOKEN` 模式：

- `X-Contract-Sign-Callback-Token`
- `X-Idempotency-Key`

`TIMESTAMP_NONCE_SIGNATURE` 模式：

- `X-Contract-Sign-Timestamp`：Unix epoch seconds。
- `X-Contract-Sign-Nonce`：供应商生成的随机串。
- `X-Contract-Sign-Signature`：`hex(HMAC_SHA256(secret, timestamp + "\n" + nonce + "\n" + canonicalPayload))`。
- `X-Idempotency-Key`：仍强制要求。

当前 `canonicalPayload` 使用后端 DTO 序列化后的 JSON，与幂等请求体 hash 保持一致。

## 4. 补偿池策略

进入补偿池：

- 回调来源已通过 Token 或签名校验。
- 回调进入业务处理后出现 `DATA_*`、`STATE_*`、`VALID_*` 异常。
- 例如：未知 `external_sign_ref`、状态值不支持、乱序回调导致状态不可推进。

不进入补偿池：

- Token 缺失/错误。
- HMAC 签名错误。
- 时间戳超出窗口。
- 幂等键缺失或幂等键冲突。

补偿任务：

- `business_type = CONTRACT_SIGN_CALLBACK`
- `compensation_type = CONTRACT_SIGN_CALLBACK_REVIEW`
- `business_id = external_sign_ref`
- `compensation_status = MANUAL_REQUIRED`
