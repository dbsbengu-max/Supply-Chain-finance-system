## 3. 核心业务流

### 3.1 兑付申请

入口：`POST /vouchers/{id}/redeem-apply`

规则：

- 仅 `ACCEPTED`、`ISSUED`、`TRANSFERRED` 可发起兑付。
- 若 `locked_amount > 0`，返回 `VOUCHER_LOCK_409`，阻止兑付。
- 发起后记录 `redeem_restore_status`、`redeem_amount`、`bpm_instance_id`。
- 凭证状态进入 `REDEEM_PENDING`。
- 写入 `REDEEM_APPLY` 流水与审计。

### 3.2 BPM 审批

业务类型：`VOUCHER_REDEEM`

规则：

- 审批前校验凭证仍处于 `REDEEM_PENDING`。
- 审批通过：凭证进入 `REDEEM_APPROVED`，写入 `REDEEM_APPROVE`。
- 审批驳回：凭证进入终态 `REJECTED`，清空 `redeem_restore_status` / `redeem_amount` / `bpm_instance_id`，写入 `REDEEM_REJECT`。
- 继承 BPM 四眼规则，发起人不能审批本人发起的兑付流程。

状态机摘要：

```text
ACCEPTED/ISSUED/TRANSFERRED
  → redeem-apply → REDEEM_PENDING
  → BPM approve  → REDEEM_APPROVED → redeem-execute → REDEEMED
  → BPM reject   → REJECTED（终态）
```

### 3.3 兑付执行

入口：`POST /vouchers/{id}/redeem-execute`

请求约束：

- 必须具备 `VOUCHER_REDEEM_EXECUTE`。
- 必须携带 `X-Idempotency-Key`。
- 必须携带有效 `X-Secondary-Auth-Token`。
- 仅 `REDEEM_APPROVED` 可执行。
- 承兑方付款账户与凭证承兑方企业一致。
- 持有人收款账户与凭证持有人企业一致。
- 账户币种必须与凭证币种一致。
- 承兑方余额不足时阻断。

执行结果：

- 承兑方账户扣减 `redeem_amount`。
- 持有人账户增加 `redeem_amount`。
- 记录兑付银行流水（`SOURCE_VOUCHER_REDEEM`）。
- 凭证可用余额归零。
- 凭证状态进入 `REDEEMED`。
- 写入 `REDEEM_PAY` 流水与 `REDEEM_EXECUTE` 审计。
- 同一幂等键重复请求返回 replay，不重复扣款。

## 5. 核验修正

本次收口过程中对齐：

- **驳回终态**：由「恢复申请前状态」改为 `REJECTED`，与列表筛选项及终态语义一致。
- **循环依赖**：`VoucherRedeemBpmCallback` 通过 `ObjectProvider<VoucherService>` 延迟注入，避免 BPM 回调与凭证服务循环依赖导致启动失败。
- **四眼测试**：自审用例由 `platform_admin` 发起并尝试自审，准确验证 `BPM_FOUR_EYES_409`（非权限 403 误报）。
- **测试编译**：`pom.xml` 补充 `junit-platform-launcher`，保障 `EA019RegressionRunner` 与 Surefire 3.x 联调。

## 7. 当前边界

当前 EA-026 仍是 MVP 闭环，以下能力建议后续补齐：

- 兑付执行未接入真实银行出金回调，当前为本地虚拟账户同步记账。
- 兑付执行未做 BPM 通过后的有效期限制。
- `REJECTED` 后是否允许重新发起兑付的业务规则需产品确认（当前未实现重开）。
- 未实现部分兑付、多次兑付、到期日自动提醒。
- 未实现兑付失败后的补偿队列和人工介入工作台。
