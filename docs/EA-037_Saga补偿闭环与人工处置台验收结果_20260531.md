## 1. 验收结论

本次交付覆盖：

- 代采 Saga 在 `ORDER_CONFIRM` 成功后、后续步骤失败时自动入队 `ORDER_ROLLBACK` 补偿任务。

- 补偿任务人工处置状态机：`CLAIM` → `SUBMIT_APPROVAL` → `APPROVE_EXECUTE`，以及 `RETRY`、`IGNORE`、`CLOSE`。
- 高风险补偿四眼审批：发起人与审批人相同返回 `BPM_FOUR_EYES_409`。
- Saga 运营 API 扩展：认领、提交审批、忽略、关闭及详情影响摘要、审计时间线。
- 前端 Saga 监控台：处置按钮、高风险标识、影响摘要、审计时间线。
- 数据库迁移 `V1_1_028`；集成测试 `SagaEa037IntegrationTest`（8 用例）纳入 `EA019RegressionRunner`。

## 4. 补偿类型与人工处置状态机

### 4.1 补偿类型（代采）

| 类型 | 触发场景 |
|---|---|
| `MARGIN_UNFREEZE` | 保证金已冻结、后续步骤失败 |
| `INVENTORY_UNFREEZE` | 库存已冻结、后续步骤失败 |
| `ORDER_ROLLBACK` | 订单已确认、后续步骤失败（EA-037 新增） |

### 4.2 人工处置流转

```text
MANUAL_REQUIRED → [CLAIM] → CLAIMED
CLAIMED → [SUBMIT_APPROVAL + reason] → APPROVED
APPROVED → [APPROVE_EXECUTE] → PENDING → PROCESSING → SUCCESS

MANUAL_REQUIRED / CLAIMED / APPROVED / FAILED → [RETRY] → PENDING（清零 retry）
MANUAL_REQUIRED / CLAIMED → [IGNORE + reason] → IGNORED
任意非终态 → [CLOSE + reason] → CLOSED（CLOSED 后不可 RETRY，409）

高风险：approve-execute 时 approver ≠ submitted_by，否则 BPM_FOUR_EYES_409
```

Worker 自动处理跳过：`CLAIMED`、`APPROVED`、`IGNORED`、`CLOSED`、`MANUAL_REQUIRED`（需人工先处置或 API 批准）。

## 8. 回归兼容调整

- **`ea027InsufficientMarginFailsAndCreatesCompensationTask`**：自有资金 Saga 先执行 `ORDER_CONFIRM` 再 `MARGIN_FREEZE`；保证金不足时除失败态外还会入队 1 条 `ORDER_ROLLBACK`，断言由 `compensationCount=0` 更新为 `1` 且存在 `ORDER_ROLLBACK` 任务（符合 EA-037 设计）。
- **`CompensationTaskProcessor.markFailure`**：转 `MANUAL_REQUIRED` 时审计日志 `operator_id` 从代采申请解析，避免 H2 非空约束失败。

## 9. 与 EA-027 / EA-028 的关系

- **EA-027**：Saga 主链路与 `MARGIN_UNFREEZE` / `INVENTORY_UNFREEZE` 入队。
- **EA-028**：Worker 自动执行、阶梯重试、`MANUAL_REQUIRED`、基础 Saga 监控台。
- **EA-037**：补齐 `ORDER_ROLLBACK`、人工认领/审批/忽略/关闭状态机、监控台处置 UI 与 8 条专项集成测试，使「失败后如何运营处理」可测试、可审计。
