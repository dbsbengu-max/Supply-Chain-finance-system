# EA-040 合同签章供应商适配层 + 单证签署状态闭环验收结果

生成日期：2026-05-31

## 1. 验收结论

**通过。** EA-040 在不接入真实电子签/区块链供应商的前提下，完成签章适配层、Mock 供应商、异步回调、失败重试补偿，并将 `sign_status` / `contract_status` 与单证中心、融资放款单证校验打通，为后续替换真实供应商预留稳定接口。

本次交付覆盖：

- 数据库迁移 `V1_1_031__contract_sign_adapter.sql`：`tr_contract_sign_task` 签署任务表；权限 `DOCUMENT_CONTRACT_SIGN`、`DOCUMENT_CONTRACT_SIGN_RETRY`。
- 供应商适配：`ContractSignProvider` 接口 + `MockContractSignProvider`（`MOCK`）+ `ContractSignProviderRegistry`。
- 编排服务：`ContractSignService`（发起签署、重试、任务查询、单证状态推进）。
- 回调服务：`ContractSignCallbackService`（`X-Contract-Sign-Callback-Token` + `X-Idempotency-Key`，与银行放款回调同模式）。
- 单证中心：采购合同登记/复核通过后进入 `PENDING_SIGN`；`/documents/validate` 对未签署合同写入 pending（`SIGN_*`）。
- 前端单证详情：展示签署状态、签署任务时间线、「发起签署」「重试签署」按钮。
- 集成测试 `ContractSignIntegrationTest`（8 用例）纳入 `EA019RegressionRunner`；全量回归 **147/147** 通过。

## 2. 状态模型

| 维度 | 取值 / 流转 |
|---|---|
| `sign_status` | `NOT_REQUIRED` → `PENDING` → `SIGNING` → `SIGNED` / `FAILED` |
| `contract_status` | `DRAFT` → `PENDING_SIGN` → `SIGNING` → `SIGNED` / `SIGN_FAILED` |
| `tr_contract_sign_task.task_status` | `SUBMITTED` → `PENDING_CALLBACK` → `SIGNED` / `FAILED` / `RETRYABLE` |

      max-retry-count: 3

### ContractSignIntegrationTest 用例摘要

1. 复核通过合同 → 发起签署 → 回调 SUCCESS → 单证 `SIGNED`
2. 回调 FAILED → 重试 → 再次回调 SUCCESS
3. 未签署合同参与 `/documents/validate` → `passed=false`
4. 未复核通过不可发起签署 → `STATE_409`
5. 签署进行中重复发起 → `STATE_409`
6. 回调 Token 缺失/错误 → `AUTH_403`
7. 回调幂等键重放 → `idempotent_replay=true`
8. 供应商提交失败 → 返回 409 且签署任务/单证失败状态落库

## 7. Codex 复核补丁

- `ContractSignService.initiateSign` / `retrySign` 对 `BusinessException` 不回滚，避免 Mock/真实供应商提交失败时失败任务、失败合同状态、复核轨迹被事务回滚。
- 新增 `ea040ProviderSubmitFailurePersistsFailureState`，覆盖 `simulate_failure=true` 的失败留痕路径。

## 8. 后续接入真实供应商建议

1. 实现 `ContractSignProvider` 并注册到 `ContractSignProviderRegistry`（如 `ESIGN_XXX`、`BLOCKCHAIN_YYY`）。
2. 保持 `external_sign_ref` 与回调体结构不变，仅替换 `createSignRequest` / 验签逻辑。
3. 生产环境可在 PostgreSQL 上补充 `external_sign_ref` 唯一约束（H2 测试库使用普通索引以兼容）。

## 9. 备注

- Mock 供应商支持 `simulate_failure=true` 用于联调提交失败路径。
- 无 JWT 的回调路径通过 `SecurityUtils.optionalCurrentUser()` 与 `AuditLogService.logAsSystem` 写入审计，避免 `AUTH_401`。
