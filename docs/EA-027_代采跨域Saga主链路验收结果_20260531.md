## 3. 核心业务流

### 3.1 审批通过 → Outbox

BPM 业务类型：`AGENCY_PURCHASE`

规则：

- 审批前校验申请处于 `REVIEWING`。
- 审批通过：申请状态 `APPROVED`，写入 Outbox 事件 `AGENCY_PURCHASE_APPROVED`。
- 幂等键：`AGENCY-PURCHASE-APPROVED-{applicationId}`。
- 审批驳回：不写入 Outbox，不启动 Saga。

### 3.2 Saga 步骤（按模式条件执行）

| 步骤 | 条件 | 动作 |
|---|---|---|
| `ORDER_CONFIRM` | 有关联 `order_id` | `SUBMITTED` 订单 → `CONFIRMED` |
| `MARGIN_FREEZE` | `SELF_FUNDED` | 冻结保证金（默认 10% 或指定 `margin_amount`） |
| `INVENTORY_FREEZE` | `STOCK_PREPARE` 且有 `inventory_id` | 扣减可用量、增加冻结量 |
| `FINANCE_CREATE` | `THIRD_PARTY_FUNDED` | 创建融资申请并回写 `finance_application_id` |

步骤记录在 `ap_agency_purchase_saga_step`，已成功步骤重复投递时跳过（幂等）。

### 3.3 失败与补偿

- 任一步骤失败：`saga_status=FAILED`，记录 `saga_last_error` 与失败步骤。
- 已完成且产生副作用的步骤入队补偿：`MARGIN_UNFREEZE`、`INVENTORY_UNFREEZE`。
- Outbox 处理器独立事务（`REQUIRES_NEW`），Saga 亦独立事务提交失败态，避免整批 Outbox 回滚。
- 保证金不足等业务异常不标记 Saga 事务 rollback-only（`noRollbackFor`）。

状态机摘要：

```text
REVIEWING → BPM approve → APPROVED + Outbox PENDING
  → Saga RUNNING → 各步骤 SUCCESS/SKIPPED → saga_status SUCCESS
  → 任一步 FAILED → saga_status FAILED + 补偿任务（如有已完成冻结步骤）
REVIEWING → BPM reject → REJECTED（无 Saga）
```

## 5. 核验修正

本次收口过程中对齐：

- **事务边界**：`AgencyPurchaseSagaService` 使用 `REQUIRES_NEW`；`AccountMarginFreezeService.freezeMargin` 对 `BusinessException` 不 rollback，保证失败态可落库。
- **Outbox 隔离**：`OutboxEventProcessor` 单事件 `REQUIRES_NEW`，Saga 业务失败时按 `saga_status` 标记 Outbox `FAILED` 而非抛异常拖垮批次。
- **循环依赖**：Saga 通过 `ObjectProvider<FinanceApplicationService>` 延迟注入。
- **BPM 四眼**：集成测试成员创建/提交，平台管理员审批。

## 7. 当前边界

- 补偿任务由 `CompensationTaskExecutor` 定时轮询（默认 30s），`CompensationTaskProcessor` 独立事务执行。
- 支持 `MARGIN_UNFREEZE`（账户解冻）与 `INVENTORY_UNFREEZE`（库存解冻）。
- 详情 API `/applications/{id}/detail` 返回 `compensation_tasks`；列表支持 `saga_status` 筛选。
- Saga 失败后订单若已 `CONFIRMED`，暂无 `ORDER_ROLLBACK` 补偿步骤。
- Outbox 失败重试策略为分钟级退避，未接入人工工作台 UI。
- 未覆盖「库存不足」「融资创建失败后的部分补偿」组合场景专项测试。
